# 答辩演示脚本 — 知行车秘

> 总时长：约 6 分钟。串词中 `【等】` 标记处稍停顿等待界面响应。

---

## Act 1: 开场（30s）

### 操作

1. 打开浏览器，进入系统 WebUI 主页
2. 展示主界面全貌：顶部状态栏（驾驶情境信息）、左侧预设快捷切换、中间对话区、底部输入框

### 串词

各位老师好，我的毕业设计是"知行车秘"——一个面向车载场景的 AI 智能体原型。

系统围绕三条数据流设计：被动记录——语音自动转录存入记忆；主动提醒——后台调度器在合适时机推送；查询响应——接收用户问题即时回复。

三个设计目标：驾驶安全优先——规则引擎硬约束不可绕过；情境感知——外部数据直接注入，不用 LLM 编造；零配置记忆——用户无需手动创建提醒，语音说了就记。

### 预期效果

- 主页正常加载，无明显报错
- 状态栏显示当前驾驶情境信息

---

## Act 2: 查询演示（90s）

> ⚠️ 切换场景预设为 🅿️ **停车场**，确认对话标签页已激活。

### 操作

1. 点击左上角场景预设下拉，选择 🅿️ **停车场**
2. 在底部输入框输入：「明天上午9点有个会议，提醒我提前出发」
3. 点击 **发送** 按钮
4. 等待数秒（LLM 推理，实际可能 5-15 秒）
5. 观察右侧四阶段面板变化

### 串词

下面演示第二条数据流——查询响应。

系统采用三阶段工作流：情境与记忆检索 → LLM 综合决策 → 格式拆解输出。每一阶段的结果都可独立审查，确保决策可解释。

核心设计原则是外部数据注入：驾驶情境不是让 LLM 自己猜的——这里是"停车场、静止"的结构化数据直接注入 Context 阶段，不依赖 LLM 内部知识。

在等待推理期间，我们可以看到右侧面板逐步更新各阶段结果。输入"明天上午9点有个会议，提醒我提前出发"——系统理解这是创建提醒的意图，会根据当前情境判断输出方式。

### 预期效果

| 面板 | 应出现内容 |
|------|-----------|
| Context | 停车场 + 静止（10min）的路况信息、相关记忆检索结果 |
| JointDecision | LLM 的完整决策文本 |
| Execution | 包含 `speakable_text` 和 `display_text` 的提醒内容 |
| 对话区 | 用户消息发送成功，系统回复显示在对话气泡中 |

### 注意事项

- 若 LLM API 响应超过 10 秒，可自然接一句「由于推理需要一点时间，我们先看下一幕，稍后回来看结果」
- 网络波动时可能返回错误提示，刷新页面重试即可

---

## Act 3: 语音演示（90s）

> ⚠️ 点击顶栏 🎙️ **语音** 标签页切换。确认麦克风已连接。

### 操作

1. 点击顶栏 **🎙️ 语音** 标签页切换
2. 确认麦克风权限已授予（浏览器弹窗点击"允许"）
3. 点击 **开始录音** 按钮
4. 用正常语速说：「提醒我下午三点去加油站，油箱快没油了」
5. 观察实时转录区域，等待文字出现
6. 点击 **停止录音**
7. 查看转录历史列表

### 串词

这是被动流——语音自动转录，零配置记忆。

流程：麦克风采集音频 → VAD 检测语音片段 → ASR 语音识别转文字 → 自动存入 MemoryBank。整个过程用户只需说话，不需要手动创建提醒，不需要填写表单。

这里用的是 sherpa-onnx 的 SenseVoice 模型，本地方案，不依赖云端 ASR，响应延迟更低。即便语音模块异常，系统的其余功能也不受影响——这是静默降级的设计。

### 预期效果

- 点击"开始录音"后按钮状态变为"录音中"（红色指示灯）
- 说话期间实时转录区域逐字显示识别结果
- 停止后历史列表新增一条转录记录
- 转录文字自动推入 MemoryBank

### 注意事项

- 若浏览器提示麦克风权限，点击"允许"
- 环境嘈杂可能影响识别准确率，建议语速稍慢、咬字清晰
- 若无可用麦克风，可跳过此幕，说明「当前环境无麦克风，语音模块支持静默降级」

> 演示至此，Act 2 的"会议提醒"和 Act 3 的"加油提醒"均已存入 MemoryBank。Act 5 主动调度的数据源已就绪。

---

## Act 4: 规则引擎演示（60s）

> ⚠️ 切换回 💬 **对话** 标签页。

### 操作

1. 选择 🅿️ **停车场** 预设（先看 LLM 自然输出）
2. 输入：「提醒我给老张回电话」
3. 点击 **发送**，观察 Execution 面板中 `channel` 字段——此时为 `visual` 或 `both`
4. 将场景预设切换为 🛣️ **高速公路**
5. 再次发送相同输入（或使用快捷指令"提醒回电"）
6. 再次观察 Execution 面板——`channel` 被规则引擎强制改为 `audio`，并附频率约束 `30min`

### 串词

这是安全优先设计的核心——规则引擎。

先用停车场场景展示 LLM 的自然输出：系统判断可以同时用文字和语音。但切换到高速公路后，7 条硬规则介入——`postprocess_decision()` 在所有输出路径上强制执行，不可绕过。高速公路场景下输出通道强制纯音频，提醒频率限制 30 分钟。

注意两次输出的对比：LLM 决策未变，但经过规则引擎后，`channel` 从 `visual`/`both` 被强行改为 `audio`。这才是安全约束的真正体现——不是 LLM 自己"懂事"，而是规则引擎兜底。

### 预期效果

- 停车场场景下：Execution 面板 `channel` 为 `visual` 或 `both`（LLM 自然输出）
- 高速公路场景下：`channel` 被强制改为 `audio`，`frequency` 字段显示 `30min` 约束
- 决策日志中可见规则引擎的修改记录

### 注意事项

- 两次发送之间确保场景预设已切换，可提一句「我把场景切到高速公路，再发一次相同内容」
- 若 LLM 在停车场场景下已输出 `audio`（概率低），仍可对比 `frequency` 字段的差异

---

## Act 5: 主动提醒演示（60s）

> ⚠️ 选择 🚦 **拥堵路段** 预设。观察右侧面板的调度器状态。

### 操作

1. 选择 🚦 **拥堵路段** 预设
2. 观察右侧面板的调度器状态和推送消息
3. 等待调度器触发（约 15-30 秒内）
4. 观察右下角 Toast 通知弹出
5. 如无自动触发，点击调度器面板的 **手动触发** 按钮

### 串词

最后一条数据流——主动提醒。此时 MemoryBank 中已有两条待办：Act 2 的"明天上午会议提醒"和 Act 3 语音存入的"下午三点加油"。当前触发的是"加油提醒"——调度器根据时间紧迫度和驾驶情境评估后自动推送。

调度器有 5 种触发源：时间事件、位置变化、情境切换、状态变更、周期轮询。当驾驶情境从停车场切换到拥堵路段，上下文变化触发主动扫描。调度器检索 MemoryBank 中的待办提醒，评估紧迫度，决定是否推送。

15 秒轮询周期加上防抖机制，避免频繁触发。提醒通过 WebSocket 实时推送，无需手动刷新。

### 预期效果

- 右下角弹出 Toast 通知，内容为"加油提醒"或"会议提醒"
- Toast 包含"接受"/"忽略"按钮
- 右侧面板显示 WebSocket 推送的消息体

### 注意事项

- 若调度器未自动触发（情境切换时间不足），使用手动触发按钮
- 可以自然串场：「为了节省时间，我直接手动触发一次」

## Act 6: 反馈学习与收尾（30s）

> ⚠️ 确认上一幕的 Toast 通知仍在屏幕右下角。

### 操作

1. 在上一步弹出的 Toast 通知中，点击 **接受** 按钮
2. 观察按钮状态变化（按钮变灰/显示已处理）
3. 如另有提醒，点击 **忽略** 演示另一种反馈

### 串词

最后演示反馈闭环。

用户对提醒的"接受"或"忽略"操作会反馈给系统——系统会根据反馈调整偏好权重，在下一次 JointDecision 时注入 LLM 提示词，影响提醒策略的调整方向。

此外，系统使用 Ebbinghaus 遗忘曲线对记忆进行衰减建模，检索时按保留率加权——长期不用的记忆自然降低检索排名。

这就是系统的三条数据流和完整交互闭环。谢谢各位老师。

### 预期效果

- 点击"接受"/"忽略"后按钮响应正常，无报错
- 可打开 MemoryBank Metrics 面板查看权重变化（可选展示）

### B. 备用方案

| 问题 | 对策 |
|------|------|
| LLM API 超时 | 刷新重试；串词自然过渡"等待时继续下一幕" |
| 语音识别偏差 | 口述正确内容说明"系统正在识别，结果稍后确认" |
| 调度器不触发 | 使用手动触发按钮 |
| 页面报错 | 刷新页面；检查终端日志定位 |
| 网络中断 | 本地方案（ASR/MemoryBank）不受影响，LLM 调用需网络连通 |
