## 一、项目概述

各位老师好，我是XXX，我的课题是《知行车秘——基于多Agent的车载AI智能体系统》。

本课题面向"车载备忘录从记录工具进化为决策与执行代理"的目标，研究并实现一个具备**情境理解、个性化学习与主动决策能力**的车载AI智能体原型系统。

---

## 二、研究背景与意义

### 2.1 问题与挑战

传统的车载备忘录多停留在事件记录与到时提醒层面，存在以下问题：

- **缺乏情境感知**：无法综合理解驾驶环境、用户状态与时空信息
- **被动响应**：只能被动响应用户请求，无法主动决策
- **交互干扰**：在复杂驾驶情境下可能造成安全隐患

### 2.2 研究意义

本系统通过多Agent协作、规则引擎约束和记忆系统设计，实现：

- **低干扰、高价值**的主动辅助
- **可解释、可控**的决策过程
- **个性化**的用户体验

## 三、系统架构

### 3.1 整体架构

```
用户输入 → Context Agent → Task Agent → Strategy Agent → Execution Agent → 提醒输出
              ↓              ↓            ↓              ↓
          历史记忆      事件抽取      安全约束       执行结果
```

### 3.2 四阶段Agent工作流

| Agent | 职责 | 输入 | 输出 |
|-------|------|------|------|
| **Context Agent** | 上下文构建 | 用户输入 + 历史记忆 + 外部上下文 | JSON上下文对象 |
| **Task Agent** | 任务理解 | 用户输入 + 上下文 | 任务类型、约束条件 |
| **Strategy Agent** | 策略决策 | 上下文 + 任务 + 安全约束 | 提醒时机、方式、内容 |
| **Execution Agent** | 执行反馈 | 决策对象 | 执行结果、event_id |

### 3.3 规则引擎（安全约束）

系统实现轻量规则引擎，基于驾驶状态自动应用安全约束：

| 规则 | 触发条件 | 约束 |
|------|----------|------|
| 高速仅音频 | scenario == "highway" | 仅音频通道，最大30分钟/次 |
| 疲劳抑制 | fatigue_level > 0.7 | 仅紧急提醒 |
| 过载延后 | workload == "overloaded" | 延后提醒 |
| 停车全通道 | scenario == "parked" | 全部通道可用 |

## 四、记忆系统

系统支持 **MemoryBank** 和 **MemoChat** 两种记忆管理策略，通过Facade工厂注册表统一管理，可根据场景灵活切换。

### 4.1 MemoryBank模式（遗忘曲线 + 分层摘要）[2]

基于Ebbinghaus遗忘曲线理论，模拟人类记忆的衰减与强化机制：

**遗忘曲线公式**：`retention = e^(-days / 5×strength)`

- 新事件初始强度为1，每次检索命中时强度+1
- 随着时间推移，记忆强度按指数衰减
- 衰减低于阈值时标记为"遗忘"（memory_strength设为0）

**三层记忆架构**：

1. **Interaction（交互层）**：原始用户查询与系统响应
2. **Event（事件层）**：语义相似的交互自动聚合为事件
   - 聚合条件：Embedding余弦相似度≥0.8 或 关键词重叠≥50%
3. **Summary（摘要层）**：按日/按总体生成层级摘要
   - 日摘要：每日事件数达到阈值自动生成
   - 总体摘要：全量事件达到阈值后生成

**个性分析**：基于每日交互记录提取用户偏好与习惯模式

**检索流程**：
1. 关键词/Embedding向量化检索 → 获取候选事件
2. 结合遗忘曲线计算保留度得分
3. 检索结果自动展开关联的原始交互记录
4. 命中事件执行记忆强化，未命中事件执行遗忘检测

### 4.2 MemoChat模式（对话缓冲 + LLM摘要）[3]

基于三阶段Pipeline的对话记忆管理：

**滚动对话缓冲**：
- 维护近期对话列表（memochat_recent_dialogs.toml）
- 触发条件：对话轮次≥10 或 总字符数>1024

**LLM主题摘要**：
- 调用LLM将对话解析为JSON格式：`{topic, summary, start, end}`
- 支持topic归类（会议/出行/采购/联系等）和NOTO（不相关）
- 摘要结果存储至memochat_memos.toml

**检索策略**：
- **FULL_LLM模式**：全量LLM精排
- **HYBRID模式**：Embedding/关键词粗筛 + LLM精排

### 4.3 反馈学习机制

用户反馈（接受/忽略）会更新策略权重：
- **accept**：对应类型权重 +0.1（上限1.0）
- **ignore**：对应类型权重 -0.1（下限0.1）

权重结果存储至strategies.toml，用于后续策略决策优化。

## 五、外部数据注入

系统支持外部上下文数据注入，用于实现情境感知能力。

**注入数据包括**：
- 驾驶员状态（情绪、工作负荷、疲劳度）
- 时空信息（当前位置、目的地、ETA、车速）
- 交通状况（拥堵等级、交通事故、延误时间）
- 驾驶场景（停车/城市道路/高速公路/拥堵）

**注入方式**：通过DrivingContextInput结构传入工作流，Context Agent直接使用注入数据构建上下文，跳过LLM推断，避免编造错误上下文。

## 六、基准测试

### 6.1 VehicleMemBench基准测试框架

VehicleMemBench是一个专门用于评估车载记忆系统性能的基准测试框架，由Chen等人在2026年提出[1]。本项目已完成适配器集成。

**框架核心特点**[1]：

1. **多用户长期记忆评估**：首个针对车载环境的多用户长期记忆基准测试
2. **可执行仿真环境**：基于23个车载工具模块构建的可执行仿真环境
3. **客观评估指标**：通过对比操作后环境状态与预设目标状态进行评估，无需LLM评分或人工评分
4. **大规模记忆事件**：每个测试样本包含超过80条历史记忆事件

**评估方法**[1]：
- 通过比较智能体操作后的环境状态与预设目标状态来判断任务成功与否
- 支持工具使用评估和记忆能力评估
- 实验表明：强大模型在直接指令任务上表现良好，但在记忆演化场景中表现不佳

**CLI执行流程**：

| 阶段 | 命令 | 功能 |
|------|------|------|
| 准备 | `prepare` | 初始化测试环境，生成测试数据 |
| 运行 | `run` | 执行基准测试，收集评估结果 |
| 报告 | `report` | 生成对比分析报告 |

**适配器实现**：
- MemoryBankAdapter：封装记忆库搜索客户端
- 支持历史文本批量导入
- 集成StoreClient统一检索接口

### 6.2 后续实验规划

基准测试框架为对照实验奠定基础，计划开展：

- **无系统基线**：不使用任何提醒系统
- **基础备忘系统**：仅支持到时提醒/静态提醒
- **完整Agent系统**：时空感知+个性化+主动策略

评估指标：任务完成率、提醒及时性、用户体验、认知负荷

## 七、存在问题和后续计划

### 7.1 存在问题

#### （1）对照实验尚未正式开展

基准测试框架已集成，但针对"无系统/基础备忘/完整Agent"三组对照实验的以下内容仍需推进：

- 实验场景设计：需要设计覆盖典型驾驶情境的测试用例
- 评估指标细化：任务完成率、提醒及时性、用户体验、认知负荷等指标的量化方法
- 志愿者招募与数据采集方案

#### （2）本地模型性能差距

基于本地小模型Qwen3.5-2B的测试显示：

- 记忆检索准确率远低于大规模模型
- 遗忘曲线参数可能需要针对小模型重新调优
- 需评估是模型规模问题还是提示词/Embedding问题

#### （3）WebUI与核心逻辑集成未完成

- 当前WebUI仅作为场景模拟和调试工具
- 无法作为完整用户交互界面投入使用
- 反馈收集、历史记录查看等用户功能待完善

#### （4）开题规划与实际实现的偏差

部分开题报告中规划的理论性研究内容在实际实现中进行了简化：

- 情境信息不完整处理：通过规则引擎和提示词模板简化
- 多模块误差传递：未实现完整的误差追溯链
- 突发事件处理机制：尚未实现

#### （5）情绪感知等高级功能暂未完成

情绪识别、生理信号融合等高级功能因数据获取和算法复杂度原因暂未实现，需要评估是否有必要性和可行性。
